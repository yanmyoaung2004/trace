# v0.2.0 Implementation Plan

## Overview

Four features to take Trace from a reactive SOAR platform to a proactive, full-cycle SOC platform.

| Feature | Est. Effort | Dependencies | Risk |
|---|---|---|---|
| 1. Case Management | 3-5 days | Investigation Manager | Medium |
| 2. EDR Connectors | 3-4 days | Response Agent, Plugin System | Low |
| 3. Automated Threat Hunting | 2-3 days | Playbook Engine, Scheduler | Low |
| 4. Kubernetes Security | 2-3 days | SIEM Decoders | Low |

**Total estimated: 10-15 days**

## Feature 1: Case Management

### Problem
Investigations are ephemeral — once completed, there's no way to add notes, attach evidence, tag IOCs, or track analyst actions. SOC analysts need a persistent case file for incident response documentation, compliance, and handoff between shifts.

### Design

```
Case
├── ID, Title, Description, Status (open/investigating/resolved/closed)
├── Severity (low/medium/high/critical)
├── Created/Updated/Closed timestamps
├── Assignee (analyst name)
├── Tags (key-value metadata)
├── Timeline (ordered list of events)
│   ├── Auto-logged (SIEM alerts, playbook steps, agent actions)
│   └── Manual entries (analyst notes, external evidence)
├── Evidence (file attachments)
│   ├── Screenshots
│   ├── Log extracts
│   └── PCAPs / files
├── IOCs (extracted indicators)
│   ├── IPv4, IPv6, Domain, URL, Hash (MD5/SHA1/SHA256)
│   └── Email address, File path, Registry key
├── Playbook executions (references to investigations)
└── Export (PDF, JSON, STIX)
```

### Implementation Steps

1. **Database Schema** (`internal/db/db.go`)
   - Add `cases` table: id, title, description, status, severity, assignee, tags (JSON), created_at, updated_at, closed_at
   - Add `case_events` table: id, case_id, type, content (JSON), source, created_at
   - Add `case_evidence` table: id, case_id, file_name, file_path, mime_type, size, created_at
   - Add `case_iocs` table: id, case_id, type, value, description, source, created_at
   - Add `case_investigations` join table: case_id, investigation_id

2. **Case Manager** (`internal/case/manager.go`)
   - `Create(ctx, title, description, severity) *Case`
   - `Get(ctx, id) *Case`
   - `List(ctx, filter) []*Case` — filter by status, severity, assignee, date range
   - `Update(ctx, id, updates) error`
   - `AddEvent(ctx, caseID, eventType, content) error`
   - `AddEvidence(ctx, caseID, filePath) error`
   - `AddIOC(ctx, caseID, iocType, value) error`
   - `LinkInvestigation(ctx, caseID, investigationID) error`
   - `ExportPDF(ctx, id) []byte`
   - `ExportJSON(ctx, id) []byte`

3. **CLI Commands** (`cmd/trace/case.go`)
   - `trace case create --title "..." --severity high`
   - `trace case list --status open --assignee @me`
   - `trace case view <id>`
   - `trace case note <id> "added context from log review"`
   - `trace case ioc <id> --type ip --value 10.0.0.5`
   - `trace case evidence <id> --file /tmp/screenshot.png`
   - `trace case export <id> --format pdf --output report.pdf`
   - `trace case close <id> --resolution "contained"`

4. **Auto-Case Creation** (`internal/siem/siem.go` + `cmd/trace/serve.go`)
   - When a SIEM alert fires with severity ≥ 4, auto-create a case
   - Link the resulting investigation to the case
   - The dispatch agent's `synthesize_report` adds the report as a case event

5. **Dashboard Case View** (`internal/server/dashboard.go`)
   - Add `/cases` route with case list table
   - Add `/cases/{id}` route with detail view (timeline, evidence, IOCs)
   - Add case status filter to main dashboard header

6. **PDF Export** (`internal/case/export.go`)
   - Use simple HTML → PDF conversion (no external dependencies)
   - Template-based: case metadata → timeline → evidence → IOCs → report
   - Serve via `trace case export <id> --format pdf`

### Security Considerations
- File upload validation (size limits, MIME type whitelist)
- Evidence files stored in `~/.trace/cases/<id>/` directory
- Export strips internal metadata (file paths, internal IPs)
- Access control via existing RBAC (viewer can view, analyst can edit, admin can delete)

### Error Handling
- Evidence file missing → skip, log warning, continue export
- Case not found → clear 404 error with suggestions
- Export failure → partial output with error annotation

### Performance
- Lazy-load timeline/evidence (not all at once for large cases)
- Pagination for IOC lists (50 per page)
- Evidence files stored on disk, not in DB (DB stores only path + metadata)

## Feature 2: EDR Connectors

### Problem
Trace's response actions (block IP, quarantine file, kill process) only work on the **local host**. In a real SOC, analysts need to respond to threats on **remote endpoints** — isolate a machine via CrowdStrike, kill a process via SentinelOne, block a hash via Microsoft Defender.

### Design

```
Trace Playbook
  ↓
EDR Agent (generic interface)
  ├── CrowdStrike Falcon
  │   ├── Get-AgentInfo(hostname) → AgentID, Status, LastSeen
  │   ├── Isolate-Host(agentID) → TaskID, Status
  │   ├── Release-Host(agentID) → TaskID, Status
  │   ├── Get-ProcessList(agentID) → []Process
  │   └── Get-NetworkConnections(agentID) → []Connection
  ├── SentinelOne
  │   ├── Get-AgentInfo(hostname) → AgentID, Status, LastSeen
  │   ├── Isolate-Host(agentID) → TaskID, Status
  │   ├── Release-Host(agentID) → TaskID, Status
  │   ├── Get-ProcessTree(agentID) → []Process
  │   └── Kill-Process(agentID, pid) → TaskID
  └── Microsoft Defender for Endpoint
      ├── Get-MachineInfo(hostname) → MachineID, Status
      ├── Isolate-Machine(machineID) → ActionID
      ├── Release-Machine(machineID) → ActionID
      ├── Run-AntivirusScan(machineID) → ActionID
      └── Block-File(machineID, fileHash) → ActionID
```

### Implementation Steps

1. **EDR Agent Interface** (`internal/edr/edr.go`)
   ```go
   type EDRClient interface {
       Name() string
       Initialize(config map[string]string) error
       GetAgentInfo(ctx, hostname) (*AgentInfo, error)
       IsolateHost(ctx, agentID) (string, error)
       ReleaseHost(ctx, agentID) (string, error)
       GetProcessList(ctx, agentID) ([]Process, error)
       KillProcess(ctx, agentID, pid) error
   }
   ```

2. **CrowdStrike Falcon Connector** (`internal/integration/crowdstrike/crowdstrike.go`)
   - API: `oauth2/token` for auth, `/sensors/queries/` for agent lookup, `/sensors/entities/` for actions
   - Rate limiting: 1,000 API calls/min (Standard tier)
   - Caching: Token cache with 30-min TTL
   - No direct file access — only API-based actions

3. **SentinelOne Connector** (`internal/integration/sentinelone/sentinelone.go`)
   - API: Token-based auth in header, `/web/api/v2.1/agents` for queries, `/web/api/v2.1/remote-commands` for actions
   - Rate limiting: 1,000/min per site token
   - Supports process tree queries (unique to S1)

4. **Microsoft Defender Connector** (`internal/integration/mde/mde.go`)
   - API: OAuth2 app registration, `/api/machines` for queries, `/api/machines/{id}/isolate` for actions
   - Requires Azure AD app registration

5. **Playbook Actions**
   - Add new response playbooks: `edr-isolate`, `edr-scan`, `edr-kill-process`
   - Each playbook takes `hostname` + optional `pid`/`file_hash`
   - The dispatch agent routes to the correct EDR based on config

6. **Support for `trace investigate` with EDR**
   - `trace investigate --playbook edr-isolate --param hostname=DESKTOP-123`
   - `trace investigate --playbook edr-scan --param hostname=DESKTOP-123`

### Rate Limiting & Resilience
- Per-provider token bucket with configurable rate
- Exponential backoff on 429 responses
- Circuit breaker: after 5 consecutive failures, back off 60s
- Graceful degradation: if EDR unreachable, log error and continue playbook with warning

### Configuration
```json
{
  "edr": {
    "crowdstrike": {
      "client_id": "abc123",
      "client_secret": "xxx",
      "base_url": "https://api.crowdstrike.com"
    },
    "sentinelone": {
      "api_token": "xxx",
      "base_url": "https://your-instance.sentinelone.net"
    },
    "mde": {
      "tenant_id": "xxx",
      "client_id": "xxx",
      "client_secret": "xxx"
    }
  }
}
```

## Feature 3: Automated Threat Hunting

### Problem
Currently, Trace is entirely reactive — it waits for an alert, then investigates. Many threats are only discovered through **proactive hunting**: "check all endpoints for this new IOC" or "scan for Log4j across the fleet" or "daily compliance report."

### Design

```
Hunt Definition
├── Name, Description
├── Schedule (cron expression: "0 */6 * * *" = every 6 hours)
├── Playbook (the playbook to execute)
├── Parameters (static: {hash: "275a..."} or dynamic: {indicator: "${last_seen_ioc}"})
├── Scope (which nodes to run on)
│   ├── "all" — all edge nodes
│   ├── "self" — local node only
│   └── "tags:linux" — nodes matching tags
├── Notify (alert on match severity ≥ threshold)
├── Status (active / paused / archived)
└── LastRun / NextRun timestamps
```

### Implementation Steps

1. **Database Schema** (`internal/db/db.go`)
   - `hunts` table: id, name, description, schedule (cron), playbook, params (JSON), scope (JSON), notify_threshold, status, last_run, next_run, created_at, updated_at

2. **Hunt Manager** (`internal/hunt/manager.go`)
   - `Create(ctx, definition) *Hunt`
   - `List(ctx, status) []*Hunt`
   - `Get(ctx, id) *Hunt`
   - `Pause(ctx, id) / Resume(ctx, id)`
   - `ExecuteNow(ctx, id) error` — manual trigger
   - `Delete(ctx, id) error`

3. **Hunt Scheduler** (`internal/hunt/scheduler.go`)
   - Runs in `trace serve` goroutine (alongside task worker)
   - Checks every 60s: `SELECT * FROM hunts WHERE status='active' AND next_run <= NOW()`
   - For each due hunt:
     a. Create investigation with intent "Hunt: {name}"
     b. Execute playbook with parameters
     c. If results contain matches ≥ threshold → create case + alert
     d. Update `last_run` and `next_run` based on cron
   - Cron parser: embedded library (no external dep needed)

4. **CLI Commands** (`cmd/trace/hunt.go`)
   - `trace hunt create --name "check-log4j" --playbook hash-lookup --param hash=... --schedule "0 */6 * * *"`
   - `trace hunt list`
   - `trace hunt view <id>`
   - `trace hunt run <id>` — execute immediately
   - `trace hunt pause <id>` / `trace hunt resume <id>`

5. **Dashboard Hunt View** (`internal/server/dashboard.go`)
   - Add `/hunts` route with hunt list + status
   - Add `/hunts/{id}` with schedule info, last run results, execution history

### Bundle Hunt Ideas (ship as defaults)

| Hunt | Schedule | Playbook | Purpose |
|---|---|---|---|
| `known-malware-scan` | Every 6h | `hash-lookup` | Scan critical paths for known malware hashes |
| `c2-domain-check` | Every 12h | `domain-reputation` | Check recent C2 intel feeds |
| `compliance-audit` | Daily | `compliance-scan` | Run CIS benchmark, report drift |
| `rootkit-sweep` | Daily | `rootkit-scan` | Global filesystem scan for rootkits |

### Safety
- Hunts never execute destructive actions (block IP, quarantine) without HITL
- Maximum concurrent hunts: 3 (configurable)
- Hunts longer than 30 minutes get a warning in logs
- `--dry-run` flag on `hunt create` validates the cron + playbook without scheduling

## Feature 4: Kubernetes Security

### Problem
Trace already has a Docker decoder. Kubernetes adds significant attack surface: RBAC abuse, privileged pods, container escape, secrets exposure, misconfigured network policies. Kubernetes audit logs provide rich security signals but require specific decoding and rule patterns.

### Design

```
Kubernetes API Server Audit Log
  ↓
Syslog / File Watcher
  ↓
K8s Audit Decoder (new)
  - Parses JSON audit events (v1.AuditEvent)
  - Extracts: user, verb, resource, namespace, response status
  - Tags: k8s, k8s-auth, k8s-config, k8s-secret
  ↓
Wazuh-style Rules (new)
  - Privileged pod creation → high severity
  - Secret access by non-service-account → high
  - ClusterRole binding modification → critical
  - kubectl exec into pod → medium
  - ConfigMap with sensitive data → medium
```

### Implementation Steps

1. **Kubernetes Audit Decoder** (`internal/siem/k8s_decoder.go`)
   - Implement `Decoder` interface
   - Parse JSON audit lines: `{"kind":"Event","apiVersion":"audit.k8s.io/v1",...}`
   - Extract fields:
     ```go
     fields["user"] = event.User.Username
     fields["verb"] = event.Verb  // create, get, update, delete, exec
     fields["resource"] = event.ObjectRef.Resource
     fields["namespace"] = event.ObjectRef.Namespace
     fields["name"] = event.ObjectRef.Name
     fields["response_code"] = event.ResponseStatus.Code
     fields["request_uri"] = event.RequestURI
     fields["source_ips"] = event.SourceIPs
     fields["impersonated_user"] = event.ImpersonatedUser.Username
     ```
   - Tags based on resource + verb:
     - `pods/exec` → `k8s:exec`
     - `secrets/get` → `k8s:secret-access`
     - `clusterrolebindings/create` → `k8s:rbac-change`
     - `pods/create` with privileged container → `k8s:privileged`
     - `namespaces/create` → `k8s:namespace-creation`

2. **Kubernetes Rules** (in `LoadDefault()` using YAML rule loader)
   - Create at least 20 rules specific to Kubernetes threats:
     - `K8S_PRIVILEGED_POD` — pod created with privileged:true or hostNetwork:true
     - `K8S_SECRET_ACCESS_ANOMALY` — secret read by non-service-account user
     - `K8S_RBAC_ESCALATION` — ClusterRole/ClusterRoleBinding modified
     - `K8S_EXEC_INTO_POD` — kubectl exec into running pod
     - `K8S_SENSITIVE_CONFIGMAP` — ConfigMap with "password" or "key" in name
     - `K8S_PERSISTENT_VOLUME_ACCESS` — unexpected PV/PVC access
     - `K8S_API_ANOMALY` — API calls from unexpected source IP

3. **K8s-Decoder YAML Rule File** (`internal/siem/rules/k8s.yaml`)
   - Convert the 20+ rules to Trace's YAML rule format
   - Load via `LoadYAMLDir()` at startup alongside built-in rules

4. **Hunt Integration**
   - Add `k8s-audit` hunt that checks for RBAC changes every hour
   - Alert on new ClusterRoleBindings or privileged pods

5. **Integration Test**
   - Test decoder with sample K8s audit log JSON
   - Verify correct tags and field extraction
   - Verify rules fire on privileged pod creation, secret access

### Log Source Configuration
```bash
# Forward K8s audit logs to Trace
kubectl create -f - <<EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: trace-audit-policy
  namespace: kube-system
data:
  policy.yaml: |
    apiVersion: audit.k8s.io/v1
    kind: Policy
    rules:
    - level: RequestResponse
      resources:
      - group: ""
        resources: ["secrets", "pods", "configmaps"]
    - level: Metadata
      resources:
      - group: "rbac.authorization.k8s.io"
        resources: ["*"]
EOF

# Then run Trace with:
./trace serve --siem --log-dir /var/log/k8s-audit
```

### Security Considerations
- K8s audit logs contain sensitive information (secrets, tokens)
- Trace stores only the fields listed above; never raw audit log body
- `raw` field in Event struct is omitted for K8s decoder (set to "")
- Audit logs should be ingested via file watcher, not syslog (TLS concern)

## Rollout Order

| Step | Feature | Why First |
|---|---|---|
| 1 | **Automated Threat Hunting** | Lowest effort, highest immediate value (makes Trace proactive) |
| 2 | **Kubernetes Security** | Standalone decoder + rules, no architectural changes |
| 3 | **EDR Connectors** | Requires case management for auto-created cases from EDR isolation |
| 4 | **Case Management** | Most complex; benefits from having hunts + EDR generating content |

## Verification

Each feature has a verification test:
- **Threat Hunting**: `trace hunt run known-malware-scan` → investigation created → results in dashboard
- **K8s**: Ingest sample audit log → evtx decoder parses → K8s rules fire → alert
- **EDR**: `trace investigate --playbook edr-isolate --param hostname=test --dry-run` → validates connection
- **Case**: `trace case create --title "test"` → `trace case view <id>` → events, IOCs, evidence displayed
